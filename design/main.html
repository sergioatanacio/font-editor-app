Diseño General - Editor Tipográfico Offline

1 Documento Maestro

Este archivo compila toda la documentación funcional y técnica del proyecto.

2 Proyecto General

2.1 Visión

Aplicación web offline, desplegable en GitHub Pages, para diseñar tipografías a partir de una plantilla vectorial.
El usuario exporta una plantilla SVG con celdas por glifo, dibuja en Illustrator/Inkscape y vuelve a importar el archivo para generar una fuente TTF sin rasterizar ni re-vectorizar.

2.2 Problema que resuelve

Los flujos tradicionales de “imagen -> vectorización automática” alteran curvas, nodos y proporciones originales.
Este proyecto preserva el vector del diseñador como fuente de verdad.

2.3 Objetivos funcionales

  1. Crear proyecto tipográfico con metadatos y métricas base.
  2. Generar plantilla editable con mapa de glifos.
  3. Importar SVG (v1) y reconocer glifos por celda.
  4. Validar integridad geométrica y reglas tipográficas.
  5. Exportar TTF.
  6. Guardar/abrir proyecto localmente sin backend.

2.4 Objetivos técnicos

  1. Arquitectura hexagonal.
  2. Dominio rico con invariantes fuertes.
  3. Máquinas de estado finitas para flujos críticos.
  4. Capa de aplicación basada en casos de uso.
  5. Infraestructura desacoplada vía puertos.
  6. Funcionamiento 100% en navegador sin servidor.

2.5 Restricciones

  1. Sin API remota ni base de datos servidor.
  2. Sin rasterización para extraer glifos.
  3. El dominio no depende de librerías externas.
  4. El formato principal de plantilla en v1 es SVG.

2.6 Decisión SVG vs PDF

Para v1 se define SVG como formato oficial de importación/exportación de plantilla por:

  1. Parseo directo de paths.
  2. Estructura textual y validable.
  3. Menor ambigüedad que PDF para identificar celdas/glifos.
  4. Mejor trazabilidad para depuración local.

PDF queda como extensión futura, condicionada a poder extraer paths vectoriales puros sin pérdida semántica.

2.7 Principios de diseño

  1. Un solo estado válido por flujo (máquinas de estado).
  2. Reglas de negocio en dominio, no en UI.
  3. Dominio orientado a intención tipográfica.
  4. Adaptadores para librerías y formatos.
  5. Errores explícitos y tipados.

3 Capa de Dominio (General)

3.1 Responsabilidad

Modelar tipografía y sus reglas invariantes:

  1. Identidad de fuente y glifos.
  2. Métricas globales y por glifo.
  3. Geometría vectorial de contornos.
  4. Reglas de mapeo Unicode.
  5. Criterios de exportabilidad.

3.2 Alcance del dominio

Incluye:

  1. Aggregate Root: Typeface.
  2. Entity principal: Glyph.
  3. Value Objects de identidad, métricas, geometría y metadata.
  4. Domain Services para validaciones transversales.
  5. Eventos de dominio (opcional en v1, diseñados desde inicio).

No incluye:

  1. Parseo de SVG/PDF.
  2. Generación binaria TTF.
  3. Persistencia de archivos.
  4. Framework UI.

3.3 Reglas estructurales

  1. Constructores privados.
  2. Fábricas create.
  3. Sin setters.
  4. Inmutabilidad por retorno de nuevas instancias.
  5. Errores tipados que extienden DomainError.
  6. Sin dependencias de infraestructura.

3.4 Estrategia de control de flujo

Se evita lógica dispersa con if anidados usando máquinas de estado en los procesos de aplicación, mientras el dominio protege invariantes del modelo.

3.5 Domain Services (criterio)

Se usa Domain Service cuando:

  1. Una regla involucra varios objetos de dominio.
  2. No pertenece naturalmente a una sola entidad/agregado.
  3. Debe permanecer libre de infraestructura.

3.6 Invariantes globales v1

  1. Typeface no permite Unicode duplicado.
  2. familyName y styleName obligatorios.
  3. unitsPerEm entero entre 16 y 16384.
  4. baseline fijo en 0.
  5. GlyphOutline con contornos cerrados (Z obligatorio).
  6. Glifos de tipo space pueden tener outline vacío.
  7. advanceWidth >= 0.

4 Dominio: Agregados y Entidades

4.1 Aggregate Root: Typeface

4.1.1 Estado

  1. id: TypefaceId
  2. metadata: TypefaceMetadata
  3. metrics: FontMetrics
  4. glyphs: ReadonlyMap<GlyphId, Glyph>

4.1.2 Métodos cerrados (v1)

  1. addGlyph(glyph: Glyph): Typeface
  2. replaceGlyph(glyph: Glyph): Typeface
  3. removeGlyph(glyphId: GlyphId): Typeface
  4. assignUnicode(glyphId: GlyphId, codePoint: UnicodeCodePoint): Typeface
  5. replaceGlyphOutline(glyphId: GlyphId, outline: GlyphOutline): Typeface
  6. changeGlyphMetrics(glyphId: GlyphId, metrics: GlyphMetrics): Typeface
  7. changeMetadata(metadata: TypefaceMetadata): Typeface
  8. changeFontMetrics(metrics: FontMetrics): Typeface
  9. validateExportReadiness(): ExportReadinessReport

4.1.3 Invariantes del agregado

  1. No duplicar GlyphId.
  2. No duplicar UnicodeCodePoint.
  3. metadata válido y no vacío en campos obligatorios.
  4. metrics válidas.
  5. Glifos requeridos por política de exportación.

4.2 Entity: Glyph

4.2.1 Estado

  1. id: GlyphId
  2. name: GlyphName
  3. unicode?: UnicodeCodePoint
  4. outline: GlyphOutline
  5. metrics: GlyphMetrics
  6. kind: GlyphKind (base | mark | ligature | space)
  7. status: GlyphStatus (empty | outlined)

4.2.2 Métodos cerrados (v1)

  1. replaceOutline(outline: GlyphOutline): Glyph
  2. clearOutline(): Glyph
  3. assignUnicode(codePoint: UnicodeCodePoint): Glyph
  4. removeUnicode(): Glyph
  5. changeMetrics(metrics: GlyphMetrics): Glyph
  6. rename(name: GlyphName): Glyph
  7. changeKind(kind: GlyphKind): Glyph
  8. markOutlined(): Glyph
  9. markEmpty(): Glyph

4.2.3 Invariantes de entidad

  1. name no vacío.
  2. Si kind !== space y status = outlined, outline no vacío.
  3. Si kind = space, outline puede ser vacío.
  4. advanceWidth >= 0.

4.3 Errores de dominio (catálogo v1)

  1. DomainError
  2. DuplicateGlyphIdError
  3. DuplicateUnicodeCodePointError
  4. GlyphNotFoundError
  5. InvalidGlyphOutlineError
  6. InvalidGlyphMetricsError
  7. InvalidFontMetricsError
  8. InvalidUnicodeCodePointError
  9. InvalidGlyphNameError
  10. InvalidTypefaceMetadataError

5 Dominio: Value Objects

5.1 Convención común

  1. Inmutables.
  2. Constructor privado + create.
  3. Método equals(other): boolean.
  4. Validación en creación.

5.2 Identidad

  1. TypefaceId
  2. GlyphId

5.3 Naming y metadata

  1. FontFamilyName
  2. FontStyleName
  3. GlyphName
  4. DesignerName
  5. FontVersion
  6. TypefaceMetadata

5.4 Mapeo de caracteres

  1. UnicodeCodePoint
  2. CharacterSet (opcional v1)

5.5 Métricas de fuente

  1. UnitsPerEm (16..16384)
  2. Ascender
  3. Descender
  4. LineGap
  5. XHeight (opcional)
  6. CapHeight (opcional)
  7. Baseline (=0 en v1)
  8. FontMetrics

5.6 Métricas de glifo

  1. AdvanceWidth (>=0)
  2. LeftSideBearing
  3. GlyphMetrics

5.7 Geometría vectorial

  1. Point
  2. PathCommand (M | L | Q | C | Z)
  3. Contour
  4. GlyphOutline
  5. BoundingBox (opcional v1)

5.8 Reglas geométricas v1

  1. Contour inicia en M.
  2. Contour termina en Z.
  3. Coordenadas finitas (sin NaN, sin Infinity).
  4. GlyphOutline acepta múltiples contornos válidos.
  5. Self-intersection no se valida en v1.

5.9 Resultado de exportabilidad

  1. ExportReadinessReport (isReady, issues)
  2. ExportIssue (code, message, glyphId?)
  3. ExportIssueCode

7 Capa de Aplicación y Casos de Uso

7.1 Responsabilidad

Orquestar flujo, transacciones lógicas y coordinación entre dominio y puertos.

No contiene reglas de negocio nucleares.

7.2 Política de flujo: Máquinas de Estado

Cada proceso complejo usa una máquina de estado finita para eliminar combinaciones inválidas.

7.3 Máquinas de estado definidas

7.3.1 1) Flujo de importación de plantilla completada

Estados:

  1. idle
  2. loadingFile
  3. parsing
  4. mapping
  5. validating
  6. applied
  7. error

Eventos:

  1. SELECT_FILE
  2. FILE_LOADED
  3. PARSE_OK
  4. PARSE_FAIL
  5. MAP_OK
  6. MAP_FAIL
  7. VALIDATION_OK
  8. VALIDATION_FAIL
  9. APPLY_OK
  10. APPLY_FAIL
  11. RESET

7.3.2 2) Flujo de exportación TTF

Estados:

  1. idle
  2. validating
  3. exporting
  4. saving
  5. done
  6. error

Eventos:

  1. EXPORT_REQUESTED
  2. VALID_OK
  3. VALID_FAIL
  4. EXPORT_OK
  5. EXPORT_FAIL
  6. SAVE_OK
  7. SAVE_FAIL
  8. RESET

7.5 Estructura típica de un caso de uso

  1. Recibe comando/DTO.
  2. Carga estado necesario vía puertos.
  3. Invoca comportamiento de dominio.
  4. Ejecuta políticas/domain services.
  5. Persiste resultado.
  6. Devuelve Result de aplicación con errores tipados de aplicación.

7.6 Domain Services usados desde aplicación

  1. GlyphOutlineValidator
  2. TypefaceExportReadinessPolicy
  3. UnicodeMappingPolicy (si aplica)

7.7 Errores de aplicación

  1. InvalidInputError
  2. ProjectNotFoundError
  3. FileReadError
  4. ImportFormatNotSupportedError
  5. UseCaseExecutionError

7.8 Regla de pureza

La capa de aplicación no manipula propiedades internas de entidades; solo invoca métodos de dominio.

8 Capa de Infraestructura

8.1 Responsabilidad

Implementar los puertos con librerías concretas y APIs del navegador, sin contaminar dominio.

8.2 Contexto de despliegue

  1. Web estática en GitHub Pages.
  2. Sin servidor propio.
  3. Toda operación en cliente.

8.3 Adaptadores previstos (v1)

  1. SvgTemplateExporterAdapter
  2. SvgGlyphVectorImporterAdapter
  3. TtfExporterAdapter (basado en opentype.js)
  4. LocalFileGatewayAdapter (File, Blob, FileReader, download)
  5. LocalProjectRepositoryAdapter (IndexedDB o localStorage)
  6. UuidIdGeneratorAdapter

8.4 Persistencia offline

Opciones:

  1. IndexedDB para proyectos grandes.
  2. localStorage solo para preferencias o estado ligero.
  3. Export/import manual de archivo de proyecto para backup.

8.5 Librerías candidatas

  1. opentype.js para exportación TTF.
  2. svg-pathdata o equivalente para parseo/normalización de paths SVG.
  3. fonteditor-core como apoyo de utilidades de fuentes (si se requiere).
  4. JSZip para exportación empaquetada (opcional).

8.6 Política de formatos

  1. Entrada oficial v1: SVG.
  2. Entrada PDF: fuera de alcance v1.
  3. Salida principal: TTF.
  4. Salidas opcionales futuras: OTF, WOFF2.

8.7 Seguridad y robustez en cliente

  1. Validar tamaño y tipo de archivo antes de parsear.
  2. Manejar errores de parseo sin bloquear UI.
  3. No ejecutar contenido activo embebido.
  4. Limitar uso de memoria en importaciones masivas.

8.8 Integración con UI

La UI consume casos de uso de aplicación.
Infraestructura no expone reglas de negocio, solo implementaciones técnicas.
